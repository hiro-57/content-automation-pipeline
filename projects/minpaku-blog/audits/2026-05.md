## 1. 技術選定

### 良い点
- Claude の prompt caching を前提に、固定知識とキーワード変動部を分けているのは妥当（`steps/claude.py:108-124`）。長い知識ベース注入には OpenAI より Claude のキャッシュ設計が合っている。
- 評価に Claude `web_search` を使う判断は、CSE や SerpAPI を別途組むより実装が軽い（`steps/evaluate.py:113-120`）。
- WordPress REST API + Application Password は標準機能で、追加プラグイン依存が少ない（`steps/wordpress.py:141-146`）。
- サムネ文字入れを画像生成 AI 任せにせず HTML/CSS で重ねる設計は正しい（`steps/thumbnail.py:175-239`）。日本語タイトルの可読性を担保しやすい。

### 問題点
- Edge headless 直叩きは Windows ローカル依存が強く、CI・Linux・サーバー実行に弱い（`steps/thumbnail.py:43-54`, `steps/thumbnail.py:220`）。Playwright の方が依存管理と再現性は上。
- Google Fonts をテンプレートで外部ロードしており、headless 撮影時にフォント読み込み完了を待つ処理が無い（`templates/thumbnail.html.tpl:7`）。サムネの文字崩れがランダムに起きる。
- Replicate Flux 固定は楽だが、画像生成失敗・品質劣化・課金上振れ時の代替プロバイダ抽象が無い（`steps/thumbnail.py:155-164`）。
- `DIFY_API_KEY` が `.env.example` に残っており、旧設計の判断がコードにも設定にも残っている（`.env.example:7-9`, `steps/dify.py:10-36`）。

### 改善案
- ブラウザレンダリングは Playwright に置き換え、フォントロード待ちとスクリーンショットのビューポート固定を明示する。
- LLM・画像生成・検索はプロバイダ境界を薄く作り、Claude/Replicate 固定を `settings` 化する。
- `steps/dify.py` と Dify 環境変数は削除するか `legacy/` に隔離し、現行運用から外す。
- サムネ生成は Flux の前にテンプレートだけで成立するフォールバック背景を用意すべき。

## 2. アーキテクチャ

### 良い点
- パイプラインは 6 ステップに分かれており、オーケストレーションと個別処理の分離は見通しが良い（`main.py:71-175`）。
- 評価・注釈・サムネ失敗時にも下書き投稿を続ける設計は、日次運用では止まりにくい（`main.py:88-157`）。
- 出力ファイルを `outputs/` に保存しており、最低限のバックアップはある（`main.py:82-111`）。
- Google Sheets のスキーマ列チェックは明確で、ヘッダー欠落時の原因が分かる（`steps/sheets.py:20-26`）。

### 問題点
- idempotency が無い。WP 投稿成功後に `mark_processed` が失敗すると、次回同じキーワードで重複下書きが作られる（`main.py:162-174`）。
- 「処理中」ステータスが無いため、複数実行すると同じ未処理行を同時に拾う（`steps/sheets.py:45-53`）。
- retry/backoff が外部 API 全般に無い。Anthropic、Replicate、WordPress、Sheets の一時障害に弱い（例: `steps/claude.py:127-138`, `steps/wordpress.py:72-80`）。
- ログが `print` だけで、構造化ログ・実行 ID・失敗ステップ・API レスポンス概要が追えない（`main.py:61-180`）。
- 評価に失敗しても投稿するため、品質ゲートとしては機能していない（`main.py:89-111`）。

### 改善案
- Sheet に `queued/running/draft_created/needs_review/failed` の状態遷移と `run_id`、`wp_post_id` を持たせる。
- WP 投稿前に同一 `run_id` または keyword slug の既存下書きを検索し、再実行時は更新にする。
- `tenacity` 等で API ごとの retry/backoff/timeout を統一する。
- `logging` に移行し、JSONL の実行ログと各ステップの入力・出力ファイルパスを残す。

## 3. コード品質

### 良い点
- 小さな関数に切れており、`generate_article`、`evaluate_article`、`annotate_article`、`create_draft_post` の役割は読み取りやすい。
- 型ヒントは主要な関数に最低限入っている（`steps/claude.py:72-77`, `steps/evaluate.py:75-81`, `steps/wordpress.py:108-114`）。
- プロンプトを `prompts/*.md` に外出しして Git 管理する方針は良い（`prompts/generate_article.md:3-4`）。
- WordPress 投稿時に Markdown を HTML 化する責務は `wordpress.py` に閉じている（`steps/wordpress.py:128-146`）。

### 問題点
- プロンプト読み込みの正規表現パーサが各ファイルに重複している（`steps/claude.py:24-45`, `steps/evaluate.py:25-44`, `steps/annotate.py:25-42`, `steps/thumbnail.py:25-40`）。
- `dict` 戻り値が多く、スキーマ保証が無い。LLM 評価 JSON の型崩れが下流にそのまま流れる（`steps/evaluate.py:136-146`）。
- `annotate_article` は「本文を 1 文字も変更しない」前提なのに、差分検証が無い（`prompts/annotate_article.md:11-20`, `steps/annotate.py:95-104`）。
- テストが見当たらない。特に JSON 抽出、タイトル分離、Sheets 状態更新、WP payload はユニットテストが必要。
- `steps/dify.py` は未使用なのに現役コードと同じ階層にあり、保守者を迷わせる（`steps/dify.py:10-36`）。

### 改善案
- `PromptTemplate`、`EvaluationResult`、`PipelineRun` などを dataclass / Pydantic で定義する。
- 共通の `load_prompt()`、`AnthropicClientFactory`、`retrying_request()` を作って重複を減らす。
- 注釈後に元本文からマーカーを除去した結果が元記事と一致するか検証する。
- `pytest` で副作用なしの関数から固め、外部 API はモック化する。

## 4. 見落としリスク

### 良い点
- WordPress 投稿は `status: draft` 固定で、自動公開しない設計（`steps/wordpress.py:133-137`）。
- `.env` と `google-credentials.json` は ignore されているように見える。
- 評価 JSON は `json_repair` で壊れた出力に一応耐えている（`steps/evaluate.py:60-72`）。
- メディアアップロードの alt text 失敗は投稿全体を止めないようにしてある（`steps/wordpress.py:89-100`）。

### 問題点
- プロジェクト直下に `.gitignore` が存在しない。親ディレクトリ依存の ignore は、移設や配布時に認証情報流出の原因になる。
- `WP_APP_PASSWORD` を Basic 認証で使っているが、`WP_SITE_URL` が HTTPS か検証していない（`steps/wordpress.py:50-52`, `steps/wordpress.py:124-126`）。
- LLM の web_search 結果を事実として評価に使うだけで、公式ソース優先や日付検証が無い（`prompts/evaluate_article.md:15-18`, `prompts/evaluate_article.md:82`）。
- `industry_facts.md` が TODO だらけなのに生成に「客観的な事実」として注入される（`knowledge/industry_facts.md:3-16`）。これはハルシネーション抑止ではなく、未整備知識の混入。
- サムネの HTML テンプレートにタイトル文字列をエスケープせず `.format()` で入れている（`steps/thumbnail.py:197-203`）。タイトルに `{}` や HTML が混じると壊れる。

### 改善案
- このプロジェクト直下に `.gitignore` を作り、`.env`、`google-credentials.json`、`outputs/`、`inbox/*.csv` を明示する。
- WP URL は HTTPS 必須チェック、認証失敗時の安全なエラーメッセージ、App Password の権限最小化を入れる。
- 法規制・統計は公式 URL、確認日、地域を構造化して KB に持たせ、記事生成時にも出典つきで使わせる。
- 投稿失敗・Sheets 更新失敗・メディアだけ残った場合の cleanup / rollback 手順をコード化する。

## 5. 目標適合

### 良い点
- 「AI は土台、人間が実体験を足す」という設計思想は正しい（`prompts/generate_article.md:18-36`）。
- 評価軸に網羅性・深さ・独自性・実用性・検索意図を入れているのは、SEO 記事の最低限の品質管理になる（`prompts/evaluate_article.md:20-31`）。
- `human_action_hints` を WP 本文中に黄色マーカーとして埋める発想は、編集作業を実行可能にしている（`prompts/annotate_article.md:24-35`）。
- `personal_experiences.md` を最重要の知識源として位置づけている点は、目標に直結している（`knowledge/README.md:13`）。

### 問題点
- 最重要の `personal_experiences.md` が例と TODO だけ。この状態では「人間超え」の源泉が存在しない（`knowledge/personal_experiences.md:10-35`）。
- `unique_perspectives.md` も TODO で、独自性を生む材料が未投入（`knowledge/unique_perspectives.md:21-25`）。
- 評価は同じ Claude が生成記事を採点する構造で、自己採点バイアスがある（`steps/claude.py:126-138`, `steps/evaluate.py:108-121`）。
- E-E-A-T を採点軸から外しており、目標が「典型的な人間ライター超え」なら一番重要な経験・信頼性をゲートしていない（`prompts/evaluate_article.md:31`）。
- 評価スコアが低くても投稿を止めないため、品質改善ループではなく「レポートを添えて流す」だけになっている（`main.py:89-111`, `main.py:162-166`）。

### 改善案
- まず運営者の実体験、費用、失敗、写真、自治体対応、実数を KB に入れる。今のコードよりデータ不足の方が致命的。
- 評価スコア閾値を設け、例えば `overall_score < 8` または `depth/originality < 7` なら WP 投稿せず `needs_rewrite` にする。
- 評価は Claude だけでなく、別モデルまたはルールベース検査で二重化する。
- 最終記事に出典、確認日、地域差、筆者プロフィールへの導線を入れる仕組みを作る。

---

**総合評価: 5.5 / 10**

最初にやるべき決定的改善は、`personal_experiences.md` と `industry_facts.md` を本物の一次情報・実体験・確認日つき事実で埋め、そのうえで低スコア記事を投稿しない品質ゲートを入れること。今の最大の弱点はコードではなく、「人間超え」を生む材料が空のまま自動化だけ先に完成している点。
