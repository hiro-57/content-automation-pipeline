# 民泊知識ベース更新ヘルパー

担当者の民泊運営経験を構造化された質問でインタビューし、`projects/minpaku-blog/knowledge/personal_experiences.md` に追記するスキル。

## なぜこのスキルが存在するか

「人間を超える品質」のブログ記事を AI で生成するには、運営者の**実体験**が知識ベースに蓄積されている必要がある。Codex の監査でも「最大の弱点はコードではなく、KB が空のまま自動化だけ完成している点」と指摘された。

しかし運営者は:

- 何を書けばいいか分からない
- 構造化が面倒で結局書かない
- 「この程度の経験、書く価値ある？」と判断できない

このスキルはそのギャップを埋める。インタビュー形式で価値ある経験を引き出し、AI 記事生成で再利用できる Markdown 形式に整えて追記する。

## 発動時の進行（7 段階）

各段階は短い質問で完結し、ユーザーの応答を待ってから次へ進む。**一度に複数の質問を束ねない**。

### Step 1 — 起点を聞く（オープン）

最初の質問:

> 最近の民泊運営で、印象に残った出来事は何ですか？些細なことでも OK です。

ユーザーが何かを語り始めたら Step 2 へ。「特に無い」と返ってきたら、以下のヒントで掘り起こす:

> よく振り返ると、こういうことはありませんでしたか？
> - ゲスト対応で困ったこと、嬉しかったこと
> - 想定外の費用が発生した
> - 自治体・業者・近隣とのやり取り
> - 開業時に書類・申請で詰まったこと
> - レビュー★1 や★5 を受けた時の理由
> - 同業者と話して気づいたこと

### Step 2 — 状況の整理（5W1H の When/Where/Who）

> それはいつ・どこで起きたことですか？関わった相手・物件は？

エピソードに文脈を与える。

### Step 3 — 何が起きたか（ナラティブ）

> 詳しく聞かせてください — 何が起きて、どう対応しましたか？

必要なら追加質問:

- どう発覚した？
- 最初に何をした？
- 結果はどうなった？

### Step 4 — 具体性の引き出し（数字・固有名詞）

ここが「人間超え」品質の核心:

> 具体的な数字を教えてください: 金額・期間・回数・%・年数 など覚えている範囲で。
> 関連する固有名詞は？ 自治体名・業者名・物件のエリア・プラットフォーム名 など。

ユーザーが概算や曖昧な記憶しか出せなくても、「約3万円」「2週間ほど」のレベルで記録する価値あり。**架空の数字を Claude 側で補完してはいけない**。

### Step 5 — 独自視点の抽出（差別化要素）

> このエピソード、他の民泊ブログでは書かれていないと思いませんか？意外だった点・想定外だった点は何でしたか？

「他では読めない情報」が記事の独自性の源泉。ユーザー自身がそれに気づいていないことが多いので、明示的に引き出す。

### Step 6 — 教訓の言語化

> 次に同じ状況になったらどうしますか？他の運営者にアドバイスするなら何と言いますか？

これがブログ記事に直接活きる「actionable な学び」になる。

### Step 7 — 整形 + カテゴリ判定 + 確認

引き出した内容を以下のテンプレートで整形:

```markdown
### {Step 2 の状況を端的に表すタイトル}

**状況**: {Step 2 の when/where/who を 1〜2 文で}

**起きたこと**: {Step 3 のナラティブを 200〜400 字程度で。Step 4 の数字・固有名詞を必ず織り込む}

**気づき・独自視点**: {Step 5 の差別化要素を 100〜200 字}

**教訓**:
- {Step 6 の学び 1}
- {Step 6 の学び 2}
- {可能なら 2〜4 個まで}
```

カテゴリは内容から自動判定する:

| キーワードの傾向 | カテゴリ（H2 セクション） |
|---|---|
| 届出・申請・開業・書類・法律・条例・自治体 | `## 開業時のトラブル` |
| ゲスト・宿泊客・レビュー・クレーム・対応・チェックイン | `## ゲストとの実際のやり取り` |
| 収益・コスト・経費・円・利益・価格・稼働率 | `## 収益・コスト` |
| 競合・同業・他物件・比較・相場・市場 | `## 競合観察` |
| 上記いずれにも該当しない | `## その他の知見`（無ければ新規作成） |

整形した結果を全文表示し、ユーザーに確認:

> 以下の内容で `knowledge/personal_experiences.md` の `{カテゴリ名}` セクション末尾に追記します。修正したい箇所があれば教えてください。

ユーザーが「OK」「これで」「いいよ」「はい」等で同意 → Step 8 へ。
「修正したい」「ここを変えて」と言われたら → 該当箇所を再質問して整形し直し、再度確認。

### Step 8 — 追記実行 + 継続確認

`Edit` ツールで `projects/minpaku-blog/knowledge/personal_experiences.md` を編集する:

1. ファイルを `Read` で読む
2. 判定したカテゴリの H2 を探す
3. その H2 セクションの末尾（次の H2 の直前）に新規エントリを追加
4. 該当 H2 が存在しない場合は、ファイル末尾に H2 ごと作成

完了したら報告:

> ✅ 追記しました。
>
> ファイル: `projects/minpaku-blog/knowledge/personal_experiences.md`
> セクション: {カテゴリ名}
>
> 他にも追加したいエピソードはありますか？

ユーザーが「ある」「もう一つ」等で続けたい意向を示したら Step 1 へ戻る。「終わり」「以上」「無い」等で終了。

## 重要な姿勢

1. **一度に 1 つの質問**: 質問を束ねない。1 ステップ 1 質問。ユーザーの返答を待ってから次へ。
2. **判断しない**: 「これは書く価値が無いのでは」と Claude が判断して飛ばさない。担当者が語った内容は記録する。
3. **誘導しない**: 答えが無ければ「該当なし」を尊重して次へ。架空の数字や名前を Claude が補完してはいけない（**ハルシネーション厳禁**）。
4. **押し付けない**: 「もっと具体的に」と何度も詰めない。ユーザーが言える範囲で記録する。深掘り質問は 1〜2 回まで。
5. **共感を返す**: 失敗談を語る相手には自然な共感を返す（「それは大変でしたね」等）。インタビュアーであって機械的な質問機ではない。
6. **長文を要求しない**: 答えが短くても OK。3 文の答えでも構造化できれば価値ある。

## 既存ファイルの構造維持

`personal_experiences.md` には初期テンプレートが書かれている。

- 既存の H2（`## 開業時のトラブル` 等）を再利用
- 冒頭の「**最重要ファイル**」等の説明は触らない
- 「例（埋め方の参考・実体験に置き換えてください）」見出しはそのまま残す（実体験エントリは対応する H2 の末尾に追加）
- 「## TODO」セクションは触らない（運営者が手動で管理する領域）

## 良い追加例

```markdown
### 京都市の住居専用地域で年間60日制限の見落とし

**状況**: 2024年3月、京都市中京区の物件で民泊新法届出を行った時。物件は築80年の町家、購入直後で改修中。

**起きたこと**: 民泊新法は年間180日まで営業可能と思い込んでいたが、京都市の条例では住居専用地域での営業が1月15日〜3月15日の60日間に制限されていた。届出時に保健所担当者から指摘されて初めて気づいた。改修費用は既に投下していたため、回収プランの再計算が必要に。

**気づき・独自視点**: 「民泊新法 = 180日まで OK」という単純な理解は危険。各自治体の上乗せ条例を**物件購入前**に確認するのが必須。京都市のように 1/3 以下に絞る自治体は他にもある。

**教訓**:
- 物件選定段階で用途地域と地方条例を確認
- 特区民泊や旅館業簡易宿所への切替を事前に検討
- 物件価格交渉時にこの制約を理由に下げ交渉できた可能性
```

このレベルの具体性 + 数字 + 教訓が入っていれば、AI 記事生成時に「人間超え」品質の素材として活きる。

## 担当者にまだファイルが見えていない場合

`personal_experiences.md` の場所を尋ねられたら:

```
projects/minpaku-blog/knowledge/personal_experiences.md
```

このファイルは `knowledge/` フォルダ全体と一緒に Git 管理されており、記事生成時に自動的に Claude に注入される。
